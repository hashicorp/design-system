# Plan Generation Prompt: Rename `--hds-*` CSS Custom Properties to `--hds-var-*`

## Context

The Helios Design System (HDS) uses two distinct namespaces for CSS custom properties:

- **`--token-*`** — Design tokens generated by `packages/tokens`. These are foundational/semantic/component-level tokens consumed as read-only values.
- **`--hds-*`** — Component-scoped CSS custom properties. These are NOT tokens. They serve different internal roles depending on context (see below).

The rename goal: **`--hds-*` → `--hds-var-*`** (only the component-scoped properties, never the `--token-*` ones).

### The three roles of `--hds-*` properties

This distinction matters because the rename is purely mechanical across all three roles, but the plan must still describe *why* each role exists and where it appears.

**Role 1: Internal SCSS intermediaries ("local CSS variables")**

Used to reduce repetition by collapsing token-derived values (size, color, spacing, etc.) into a single property declared and consumed within the same SCSS file. There is no JS involvement.

Example from `accordion.scss`:
```scss
// Declared once per size variant, resolved from --token-* values:
.hds-accordion-item--size-small {
  --hds-accordion-item-toggle-gap: var(--token-accordion-item-toggle-gap-small);
  --hds-accordion-item-toggle-padding-top: var(--token-accordion-item-toggle-padding-top-small);
  // ...
}

// Consumed later in the same file:
.hds-accordion-item__toggle {
  gap: var(--hds-accordion-item-toggle-gap);
  padding: var(--hds-accordion-item-toggle-padding-top) ...;
}
```

These variables are internal implementation details and are not documented as public API (consumers can technically override them, as with any CSS custom property but that's irrelevant).

**Role 2: JS-to-CSS bridge properties**

Used to pass runtime values from Ember args into CSS through JavaScript/Glimmer templates. In most components, consumers set Ember `@arg` values, and component code maps them to `--hds-*` properties via inline styles (TypeScript or Handlebars `{{style}}`) consumed in SCSS.

Example from `layout/grid/index.gts` + `layout/grid.scss`:
```typescript
// Set in JS:
inlineStyles['--hds-layout-grid-column-min-width'] = someValue;
```
```scss
// Read in SCSS:
grid-template-columns: repeat(
  var(--hds-layout-grid-column-fill-type),
  minmax(var(--hds-layout-grid-column-min-width), 1fr)
);
```

**Role 3: Public API / consumer-readable properties**

A very small subset of `--hds-*` variables is documented for direct consumer override in CSS: `--hds-layout-flex-row-gap`, `--hds-layout-flex-column-gap`, `--hds-layout-grid-row-gap`, and `--hds-layout-grid-column-gap`.

Example in consumer app CSS:
```scss
.my-flex-layout {
  --hds-layout-flex-row-gap: 20px;
  --hds-layout-flex-column-gap: 12px;
}
```

These are the `--hds-*` variables with the highest migration impact for direct overrides. In practice, consumer codebases also use many `--hds-*` values as read-only references (for example inside `var(--hds-...)`) without overriding them.

---

## Goal

Produce a **reusable, codebase-agnostic automation** that renames all `--hds-*` CSS custom properties to `--hds-var-*` in any target Ember codebase. The rename is purely mechanical, so the automation must run against any project. The HDS monorepo is only the first test case for development and validation (see *HDS Monorepo: Pilot and Validation Context*), with no special status.

---

## Requirements and Constraints

### The rename rule
- **Pattern:** `--hds-` → `--hds-var-`
- **Applies to:** all CSS custom property names (both declarations and usages)
- **Does NOT apply to:** `--token-*` variables (different namespace, never touched)
- **Regex pattern:** `(--hds-)([a-zA-Z][a-zA-Z0-9_-]*)` → `--hds-var-$2`

### Safety constraints
- Rename must be purely mechanical — no semantic changes, no logic changes.
- Test suite must pass after the rename.

### Target environments for the automation

Target codebases are Ember projects (the HDS monorepo and consumer apps built with it). File types in scope:
- CSS and SCSS (`.css`, `.scss`)
- JavaScript and TypeScript (`.js`, `.ts`)
- Glimmer single-file components (`.gts`)
- Handlebars templates (`.hbs`)

## Deliverables to Plan

The plan should propose a single **recommended migration toolkit** that can be reused across Ember codebases. The toolkit should be layered as follows:

### Recommended approach
- A deterministic CSS/SCSS rename tool for stylesheet files.
- A deterministic codemod for JS/TS/GTS/HBS files.
- A small orchestration layer that runs both in sequence and reports remaining matches.
- A reusable Copilot/AI skill (or equivalent agent-instructions artifact) as an additional layer:
  - primarily to verify that the deterministic tools left no `--hds-*` usages behind,
  - and secondarily as the main driver in codebases where the deterministic tools cannot be run.

The plan should assume this layered approach is the default target outcome:
- deterministic transforms first,
- AI verification second,
- AI-driven fallback only when deterministic tooling is not feasible.

Expected deliverables:
- A CSS/SCSS rename tool suitable for reuse across Ember codebases.
- A `packages/codemods` codemod for JS/TS/GTS/HBS usage sites.
- A runner/orchestration script for executing the migration end-to-end.
- A reusable AI skill (or equivalent agent-instructions artifact) for verification/fallback use.

## Plan Structure Expected

The plan to be generated should include:

1. **Pre-conditions / audit step** — confirm there are no `--hds-var-*` conflicts already in the target codebase before running.
2. **Implementation steps** for each chosen automation artifact, including:
   - File location (where to create/place the artifact)
   - Input/output contract
  - Edge cases to handle (escaped strings, multiline values, test assertions)
3. **Execution order** — SCSS sources → GTS/TS sources → template and doc files → rebuild and test. When running against the HDS monorepo pilot, this maps to: `packages/components/src/styles/` → `packages/components/src/components/` + `showcase/tests/` → `showcase/app/` + `website/` files.
4. **Testing strategy** — how to verify correctness (diff output, test suite, visual regression).
5. **Verification / fallback strategy** — how the AI coding agent should be used to check for leftovers and when it should act as the primary driver instead of the deterministic tools.

Include a post-migration validation step for JS-to-CSS bridge variables to ensure the renamed custom property names still match between assignment sites in JS/GTS and consumption sites in SCSS.

The plan may keep the exact packaging of the AI layer as an implementation detail, but should state whether a reusable skill is preferable to a one-off prompt and outline the minimum responsibilities that AI layer must cover.

The plan may also recommend phased or sub-agent execution if repository size, context limits, or runtime make a single-agent workflow impractical.

### Affected File Types and Usage Patterns

#### 1. SCSS files (`*.scss`)

Two roles:

**a) Declaration (definition of the custom property with a default value):**
```scss
// In a component's .scss file
.hds-layout-flex {
  --hds-layout-flex-row-gap: 0px;
  --hds-layout-flex-column-gap: 0px;
  gap: var(--hds-layout-flex-row-gap) var(--hds-layout-flex-column-gap);
}
```

**b) Usage via `var()`:**
```scss
gap: var(--hds-layout-flex-row-gap) var(--hds-layout-flex-column-gap);
```

Note: some declarations are inside Sass interpolations (used to build responsive variant names):
```scss
--hds-layout-grid-column-min-width: var(--hds-layout-grid-column-width-#{$view});
```
These interpolated suffixes (`#{$view}`) are dynamic; the rename stays mechanical and only changes the `--hds-` prefix.

#### 2. GTS/TypeScript files (`*.gts`, `*.ts`)

Three sub-patterns:

**a) TypeScript type definitions (accepted CSS custom property API):**
```typescript
'--hds-layout-grid-column-min-width'?: string;
'--hds-layout-grid-column-fill-type'?: string;
'--hds-layout-grid-column-width-sm'?: string;
```

**b) Object key assignments (setting CSS custom properties as inline styles via JS):**
```typescript
inlineStyles['--hds-layout-grid-column-min-width'] = someValue;
styles['--hds-form-super-select-dropdown-max-width'] = '40em';
```

**c) Handlebars `{{style}}` helper in GTS templates:**
```hbs
{{style --hds-key-value-inputs-columns=this._gridTemplateColumns}}
```

#### 3. HBS template files (`*.hbs`) — website docs/code snippets

Inline style usage shown in code examples:
```hbs
<Hds::Layout::Flex @gap="24" style="--hds-layout-flex-column-gap: 20px">
```

#### 4. Test files (`*.gts` in `showcase/tests/`)

`hasStyle` assertions checking that the correct CSS custom property is set:
```typescript
assert.dom(el).hasStyle({ '--hds-key-value-inputs-column-index': '2' });
assert.dom(el).hasStyle({ '--hds-form-super-select-dropdown-max-width': '40em' });
```

Also string-form style assertions:
```typescript
"--hds-form-super-select-dropdown-max-width: 40em; --hds-form-super-select-selected-text-display: none;"
```

## Additional Notes for Plan Author

- The HDS monorepo already has `packages/codemods` (jscodeshift-based). New transforms should follow conventions there.
- The HDS SCSS instruction file at `.github/instructions/languages/scss.instructions.md` defines code style constraints for SCSS edits.
- Do NOT rename Sass variables (`$hds-something`) — only CSS custom properties (`--hds-something`).
- The documented direct-override `--hds-*` variables are limited to layout gap overrides: `--hds-layout-flex-row-gap`, `--hds-layout-flex-column-gap`, `--hds-layout-grid-row-gap`, and `--hds-layout-grid-column-gap`. Consumer codebases also contain read-only `var(--hds-...)` usages; both patterns must be included in migration guidance.
- The `code-editor/theme.scss` exists in `dist/` but not in `src/` — verify whether it is auto-generated before touching it.
